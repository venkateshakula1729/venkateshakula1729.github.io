The ML Systems Stack: From Data Pipelines to Inference

Published: March 15, 2026 Updated: Apr 12, 2026 ⏱ 15 min read

Most ML courses teach you to train models. Almost none teach you to run them in production. After working on HPC systems (CS633), operating systems (CS330), and multi-agent LLM pipelines (Adobe Research), I have developed opinions about what the full ML systems stack should look like. This post is an opinionated walkthrough.

Layer 1: Data Ingestion

Everything starts with data. The first decision — batch vs. streaming — determines the architecture of everything downstream.

  • Batch processing is simpler and sufficient for most training workloads. Tools: Apache Spark, Dask, or even well-written Python scripts for moderate data sizes.
  • Stream processing is necessary when features depend on real-time events — trading signals, user behavior, sensor readings. Tools: Kafka + Flink, or lighter-weight solutions like Redis Streams.

The critical insight from my HPC work: I/O is almost always the bottleneck. In CS633, replacing sequential I/O with parallel MPI-IO gave us 8× speedup. The same principle applies to data pipelines — parallelize reads, use columnar formats (Parquet, Arrow), and never let your GPU sit idle waiting for data.

Layer 2: Feature Engineering

The gap between academic ML and production ML is largest at the feature engineering layer. In courses, features are given. In production, they must be computed, versioned, validated, and served at inference time with the same code that produced training features.

Training-serving skew — where features are computed differently during training vs. inference — is the single most common source of silent model failures.

Layer 3: Training Infrastructure

For single-GPU training, PyTorch is sufficient. For distributed training, the stack becomes complex:

  • Data parallelism (DDP) — each GPU trains on a different data shard, gradients are all-reduced
  • Model parallelism — for models too large for a single GPU, split layers across devices
  • Pipeline parallelism — overlap computation across micro-batches to improve GPU utilization

From CS633, I learned that the communication pattern matters as much as the computation. Ring all-reduce, tree reduction, and gradient compression are not academic curiosities — they determine whether your training job runs in hours or days.

Layer 4: Serving and Inference

A model that cannot serve predictions at the required latency and throughput is useless. Key considerations:

  • Batching — dynamic batching (collecting multiple requests and processing them together) is essential for GPU utilization
  • Quantization — INT8/FP16 inference can halve latency with minimal accuracy loss for most models
  • Caching — for models with repeated inputs (recommendation systems, search), caching predictions is often more efficient than recomputing

Layer 5: Observability

This is the layer most teams skip. A model in production without monitoring is a ticking time bomb. You need:

  • Input distribution monitoring — detect data drift before it causes silent accuracy degradation
  • Prediction distribution monitoring — sudden shifts in output distribution signal model issues
  • Latency monitoring — p50, p95, p99 latencies, not just averages
  • Business metric correlation — link model predictions to downstream outcomes

What Quantitative Systems Add

In quantitative finance, the ML stack has additional constraints that make everything harder:

  • Latency requirements are measured in microseconds, not milliseconds — this rules out most standard serving frameworks
  • Non-stationarity — financial data distributions shift constantly, requiring continuous retraining and model selection
  • Adversarial environment — other market participants adapt to your signals, creating a moving target
  • Regulatory constraints — explainability, audit trails, and risk limits impose architectural requirements

Building ML systems is engineering work that requires both breadth and depth. If this space interests you, start with the fundamentals: operating systems (how memory and I/O actually work), networking (how data moves between machines), and distributed systems (how machines coordinate). The ML-specific layers build on top of these foundations.
