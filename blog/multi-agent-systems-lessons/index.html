Building Efficient Multi-Agent Systems: Lessons from Adobe Research

Published: April 10, 2026 Updated: May 1, 2026 ⏱ 18 min read

During summer 2025, I worked at Adobe Research's Big Data Experience Lab on a problem that sounds deceptively simple: make multi-agent LLM systems communicate more efficiently. Three months later, we had achieved a 35% reduction in decode tokens with 1.5× latency improvements while maintaining accuracy on three benchmark suites. Here is what I learned.

The Problem is Communication, Not Intelligence

When most people think about improving multi-agent LLM systems, they focus on making individual agents smarter — better prompts, better models, better reasoning chains. But after systematic benchmarking of AutoGen, LangChain, DSPy, TextGrad, and AdalFlow, I found that the dominant bottleneck was not agent intelligence. It was inter-agent communication.

Agents were sending each other entire conversation histories, repeating context that the receiving agent already had, and filling their context windows with information irrelevant to their specific subtask. The result: token waste, context pollution, and compounding latency.

Three Key Insights

1. Context Compression Must Be Task-Aware

Naive approaches to reducing communication — summarization, truncation, random sampling — all fail for the same reason: they do not know what matters for the downstream task. A summary that preserves the "main idea" might discard the specific data point that the next agent needs to make a decision.

Our solution was to make the compression module aware of the task decomposition. Each agent receives only the context slice relevant to its assigned subtask.

2. Real-Time Coordination is Feasible

There is a common assumption that large language models are too slow for real-time coordination. In practice, with structured communication protocols — where agents exchange compact, typed messages rather than free-form text — coordination overhead becomes manageable.

The key is designing the protocol so that agents do not need to wait for complete responses. Partial results can be streamed and consumed incrementally.

3. Evaluation Must Be Multi-Dimensional

We validated on HotpotQA (multi-hop QA), GPQA (graduate-level reasoning), and GSM8K (mathematical problem-solving). No single benchmark captures the full picture. A system that excels at QA might fail at code generation. Multi-dimensional evaluation is not a luxury — it is a requirement.

What I Would Do Differently

  • Start with profiling, not optimization. We spent the first two weeks running experiments without a clear bottleneck analysis. The systematic profiling phase, when we finally did it, revealed that 60% of tokens were redundant — and pointed us directly at the right intervention
  • Build evaluation infrastructure early. Ad-hoc evaluation slows everything down. Automating benchmark runs from day one would have saved weeks
  • Read more broadly. The most useful insight — task-aware routing — came from the distributed systems literature, not the LLM literature. Cross-pollination between fields is underrated

Connections to Quantitative Systems

Many of these lessons transfer directly to quantitative trading systems:

  • Communication overhead between components (data feeds, signal generators, risk managers, execution engines) is a constant concern
  • Task-aware routing is analogous to routing market data only to the strategies that need it
  • Multi-dimensional evaluation mirrors the need to evaluate strategies across multiple market regimes
