Building GemOS

Aug – Nov 2024IIT KanpurGitHub ↗
[ Hero Image — Memory layout / page table visualization ]

Overview

GemOS is an educational operating system used in IIT Kanpur's CS330 course. Over a semester, I implemented core OS components from scratch — from user-space tools (grep, syscall tracers) to kernel-level memory management (custom allocators, page table manipulation) — building a ground-up understanding of how systems software works.

Motivation

Systems programming is where theory meets hardware constraints. Understanding how malloc actually works, how page tables translate virtual to physical addresses, and how processes coordinate via pipes and signals — these are foundational for anyone working on performance-critical systems, whether in HPC, trading infrastructure, or ML serving.

Technical Details

User-Space Tools

  • grep-like tool — pattern matching on file streams using efficient string algorithms
  • Syscall tracer — used strace internals to intercept and log system calls from target processes
  • Process-chaining shell — built using fork(), exec(), and pipe() to compose arbitrary process pipelines

Memory Management

  • Custom malloc() and free() — implemented a dynamic memory allocator using a free-list approach with linked lists, handling fragmentation through coalescing adjacent free blocks
  • mremap() syscall — implemented virtual memory remapping by allocating new pages, copying contents, updating page table entries, and handling multi-level page table walks

Results

All components passed the course's automated test suite. The custom allocator handled allocation patterns ranging from small uniform allocations to adversarial fragmentation-heavy workloads. The mremap implementation correctly handled multi-level page table traversal for various virtual address ranges.

Lessons Learned

  • Memory management is fundamentally about tradeoffs — time vs space, internal vs external fragmentation, simplicity vs performance
  • fork() is conceptually elegant but expensive — copy-on-write is not just an optimization, it is essential for practical process creation
  • Page table walks are where virtual memory abstraction meets hardware reality — understanding this is critical for performance-sensitive systems work
  • Debugging kernel code requires fundamentally different tools and mindset than user-space programming

Future Work

  • Implementing a slab allocator for kernel-object caching
  • Adding a simple virtual file system (VFS) layer
  • Exploring lock-free data structures for concurrent kernel operations
